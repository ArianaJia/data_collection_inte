## 2026-07-14

### 调度架构重构
- 任务与队列全部切换为**静态分配**（StaticTask_t / StaticQueue_t），堆可用空间从 ~5KB 恢复至 ~15KB。
- 任务重命名，直接反映业务职责：
  - `myTask02` → `CanForViehcle`（整车 CANB 解码）
  - `myTask03` → `CanForBMS`（电池箱 CAN 解码）
  - `myTask04` → `MLX90640`（热成像采集）
  - `myTask05` → `TensionSensor`（RS485 拉力传感器）
  - `myTask06` → `SPICanControl`（SPI1/MCP2518FD CANA 轮询）
  - `myTask07` → `SPICanCDC`（SPI3/MCP2518FD CDC 监视）
  - `myTask08` → `Publish4G`（4G publish + USART1 发送）
  - `myTask09` → `CanBText`（CAN1 TX 发送）
- 新增 `initTaskBoot`（osPriorityLow1）：**集中完成所有模块初始化后自销毁**，初始化顺序为 MLX90640 → MCP2518FD-A → MCP2518FD-B → 4G AT 握手。各运行时任务不再自行初始化，仅检查就绪标志（`FDCAN_IsReady` / `MLX90640_App_IsSensorReady`）。
- 4G AT bootstrap **从 `MX_USART3_UART_Init` 前调度器阶段移除**，改为 `initTaskBoot` 中执行，消除启动时 15-20 秒阻塞。

### 函数命名规范化
- 所有内部辅助函数去掉 `Task0X_` 前缀，改为模块前缀：`CanVehicle_Dispatch`、`CanBattery_Dispatch`、`Mlx_SendSummary`、`Rs485_StartDma`、`SpiCana_Poll`、`SpiCdc_Poll` 等。

### 4G 模块通信优化
- USART1 AT 命令发送从 `HAL_UART_Transmit`（阻塞）改为 `HAL_UART_Transmit_DMA`（DMA + osDelay 轮询），不再忙等 CPU。
- `HAL_Delay` 全部替换为 `osDelay`，bootstrap 期间正常释放 CPU 给其他任务。

### Bug 修复
- AT bootstrap 全程零 HAL 阻塞：
  - 发送端：`HAL_UART_Transmit` → `Y100M_Uart1DmaSend`（USART1_TX_DMA + osDelay 轮询）
  - 接收端：逐字节 `HAL_UART_Receive` → `HAL_UARTEx_ReceiveToIdle_DMA`（USART1_RX_DMA + idle 回调）
  - 延时：`HAL_Delay` → `Y100M_DelayMs`（调度器运行后用 osDelay 让出 CPU）
  - `usart.c` 补 `#include "cmsis_os2.h"`（`osKernelRunning` 声明缺失导致编译报错）
- 查明确认 4G 模块 RTS/CTS 接线错误后，USART1 关闭硬件流控（`HwFlowCtl = DISABLE`）。

### 追加修复（同日）
- **USART DMA 完成问题**：`HAL_UART_Transmit_DMA` 内部在 DMA stream 传完后开启 UART TC 中断等移位寄存器清空，但 NVIC 上 USART1/USART3 中断未使能，导致 `UART_EndTransmit_IT` 永不执行、`gState` 永远不回 READY、第二条 DMA 发送死等。修复：`usart.c` 添加 `USART1_IRQHandler` + `USART3_IRQHandler`（调用 `HAL_UART_IRQHandler`），并在 `USARTx_MspInit` 中使能 NVIC。发送 buffer 补 `length=0` 清零防御。
- **I2C 过度设计**：`TCA9548A_WriteChannelMask`、`MLX90640_I2CWrite`、`MLX90640_I2CGeneralReset` 三处对 1-2 字节操作使用 DMA 异步传输，但 I2C/DMA IRQ 优先级（5）踩在 FreeRTOS `configMAX_SYSCALL` 边界上，`taskENTER_CRITICAL` 可直接屏蔽。改为阻塞模式 `HAL_I2C_Master_Transmit` / `HAL_I2C_Mem_Write`，仅保留 `MLX90640_I2CRead`（832 字 EEPROM 大数据）继续用 DMA。
- **MLX90640 ServiceHealth 循环 bug**：`return status` 错误写在 for 循环内部，sensor 0 TCA9548A 失败后立即退出，后续传感器永远不被尝试。移到循环外。
- **CAN 未启动**：`CAN_Start()`（启动 hcan1/hcan2 + 使能 RX 中断）从未被调用。加入 `initTaskBoot` 初始化第一步。
- **initTaskBoot 进度日志**：初始化的每个阶段前输出调试信息，便于定位卡死位置。

## 2026-07-16

### MLX90640 四路热成像链路修复
- 对照已验证的 F103 `MLX90640_test` 工程，收敛当前工程的 MLX90640 业务层流程：初始化阶段完成四个 TCA9548A 通道的 probe、EEPROM dump、参数提取、Chess mode 和刷新率配置；采集阶段不再反复读取 EEPROM。
- 恢复参考工程中的 I2C1 总线恢复逻辑：上电首次选择 TCA9548A 通道前，手动释放 PB6/PB7、输出 9 个 SCL 脉冲并关闭所有 TCA 通道；连续 BUSY/TIMEOUT 后重新恢复总线并重选通道。
- 修正 MLX90640 HAL 地址定义，统一使用 `(0x33 << 1)`，避免 7-bit/8-bit 地址混用导致通信行为依赖 include 顺序。
- 保留 MLX90640 大块读取使用 I2C DMA，但在 FreeRTOS 运行后 DMA 等待循环中加入 `osDelay(1)`，避免高优先级初始化任务忙等饿死 defaultTask、调试日志与 LED 闪烁。
- 扩大 `initTaskBoot` 和 `MLX90640` 任务栈，降低 `ExtractParameters`、`CalculateTo` 等浮点/数组计算阶段栈溢出风险。

### 多传感器帧采集策略调整
- 为四个 MLX90640 分别静态缓存 `paramsMLX90640` 参数；EEPROM 原始 dump 仅保留一份静态 scratch，初始化时逐通道读取并立即提取参数，兼顾 F4 RAM 余量和运行期效率。
- 为每个传感器维护独立温度图缓存和 subpage mask。采集时每次只读取一个 ready subpage，跨多次轮询累计 `0x01/0x02`，凑齐 `0x03` 后再执行坏点修正、区域均值更新和帧计数递增。
- 新增 `MLX90640_FRAME_NOT_READY_ERROR` 与 `MLX90640_FRAME_INCOMPLETE_ERROR` 的业务层区分：未 ready 或未凑齐双 subpage 不再作为硬错误刷屏，任务层只对真正 I2C/HAL/数据错误输出 capture error。
- 为 `MLX90640_GetFrameData` 增加调试读数接口，记录最后一次状态寄存器、frame/aux 校验失败 index 和失败值，便于后续定位 `0x7FFF` 帧数据问题。
