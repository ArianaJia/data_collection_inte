## 2026-07-16

### 双 CAN 映射与硬件滤波修正
- 明确并修正当前板级实际接线：`CAN1 = 电池箱总线`，`CAN2 = 整车总线`。此前代码中 CAN1/CAN2 业务归属与物理连接相反，导致用整车标准帧 `0x501` 验证时，误以为 CAN1 接收链路异常；修正后，整车相关接收/发送、调试 LED 和诊断日志均统一落在 CAN2 路径。
- 将统一接收回调中的总线分发同步翻转：CAN1 进入 `CAN_CHANNEL_BATTERY_BOX`，CAN2 进入 `CAN_CHANNEL_VEHICLE_CANB`；`PD13` 调试翻转也移动到 CAN2 接收分支，便于直接观察整车总线中断是否命中。
- 重构 CAN 发送任务和队列命名，消除“名字写 CAN1，实际发 CAN2”的误导：`CanBTextTask` 更名为 `VehicleCanTxTask`，`CAN_Tx_QueueHandle` 更名为 `VehicleCanTxQueueHandle`，`freertos_can1_tx_queue_put` 更名为 `freertos_vehicle_can_tx_queue_put`，发送函数统一按“整车 CAN 发送链路”语义命名。
- 为 CAN1/CAN2 拆分独立上下文：不再共用单个 `g_can_msg` scratchpad，而是分别维护 `g_can1_ctx` / `g_can2_ctx` 的收发头、邮箱号、接收缓存和运行时状态，避免中断与任务之间互相覆盖状态。
- 修正接收回调结构：删除无效的 `HAL_CAN2_RxFifo0MsgPendingCallback`，统一使用 HAL 标准 `HAL_CAN_RxFifo0MsgPendingCallback`，并同时处理 CAN1/CAN2；同时将 `HAL_CAN_GetRxMessage` 的 FIFO 参数从语义错误的 `CAN_FILTER_FIFO0` 规范化为 `CAN_RX_FIFO0`。
- 去掉“接收异常直接死机”的危险路径：`HAL_CAN_GetRxMessage()` 失败不再进入 `Error_Handler()`，而是记录错误码并直接返回，避免偶发 CAN 接收异常把整个系统永久锁死。
- 增加 CAN 启动、收发诊断上下文：记录 `start_status`、`notify_status`、`last_tx_status`、`last_tx_error`、邮箱空闲数、收发计数等，便于定位“消息未发出”究竟是队列未入、邮箱占满、总线无 ACK 还是未真正启动。
- 针对发送邮箱占满加入最小恢复逻辑：当 `HAL_CAN_AddTxMessage()` 前检测到邮箱空闲数为 0，或发送失败且邮箱已满时，主动 `HAL_CAN_AbortTxRequest()` 中止挂起邮箱并清理错误状态，避免历史挂起帧永久占死 3 个发送邮箱。
- 调试期间为 CAN2 接收链路补充了 `CAN2 RX ide/std/ext/dlc/data` 串口日志，实测证明在“全接收”配置下，整车标准帧 `0x501`、`0x529` 等均可被 MCU 正确读头和进入回调；目前该诊断打印已改为默认注释，仅保留接口，后续需要时可再临时打开。
- 关键定位结论：此前看似“只有 `0x00` 能触发中断”，根因并非总线或 `HAL_CAN_GetRxMessage()` 读头错误，而是 CAN2 硬件滤波 bank 误用了 `<14` 的区间，导致双 bxCAN 架构下滤波根本未按 CAN2 归属生效。修正后确认：**CAN2 的过滤器编号必须配置为 `>=14`**，CAN1 使用 `0~13`。
- 先将 CAN1/CAN2 临时切回“全接收”完成收发链路闭环验证，确认整车总线与回调路径正常后，再恢复硬件滤波：CAN2 采用数组驱动的标准帧 `16-bit IDLIST` 配置，并将所有 bank 放到 `14+` 区间；CAN1 改为基于电池箱宏定义数组的 `32-bit IDLIST` 配置，使用 `0~13` 区间，按每 2 个 ID 占 1 个 bank 的方式装载。
- 这次双 CAN 滤波排查过程中，重点参考了以下经验文章，尤其是“CAN2 的过滤器编号需配置为 `>=14`”这一双 CAN 易踩坑点：`https://blog.csdn.net/m0_50560976/article/details/127557947`

### MLX90640 四路热成像链路修复
- 对照已验证的 F103 `MLX90640_test` 工程，收敛当前工程的 MLX90640 业务层流程：初始化阶段完成四个 TCA9548A 通道的 probe、EEPROM dump、参数提取、Chess mode 和刷新率配置；采集阶段不再反复读取 EEPROM。
- 恢复参考工程中的 I2C1 总线恢复逻辑：上电首次选择 TCA9548A 通道前，手动释放 PB6/PB7、输出 9 个 SCL 脉冲并关闭所有 TCA 通道；连续 BUSY/TIMEOUT 后重新恢复总线并重选通道。
- 修正 MLX90640 HAL 地址定义，统一使用 `(0x33 << 1)`，避免 7-bit/8-bit 地址混用导致通信行为依赖 include 顺序。
- 保留 MLX90640 大块读取使用 I2C DMA，但在 FreeRTOS 运行后 DMA 等待循环中加入 `osDelay(1)`，避免高优先级初始化任务忙等饿死 defaultTask、调试日志与 LED 闪烁。
- 扩大 `initTaskBoot` 和 `MLX90640` 任务栈，降低 `ExtractParameters`、`CalculateTo` 等浮点/数组计算阶段栈溢出风险。

### 多传感器帧采集策略调整
- 为四个 MLX90640 分别静态缓存 `paramsMLX90640` 参数；EEPROM 原始 dump 仅保留一份静态 scratch，初始化时逐通道读取并立即提取参数，兼顾 F4 RAM 余量和运行期效率。
- 为每个传感器维护独立温度图缓存和 subpage mask。采集时每次只读取一个 ready subpage，跨多次轮询累计 `0x01/0x02`，凑齐 `0x03` 后再执行坏点修正、区域均值更新和帧计数递增。
- 新增 `MLX90640_FRAME_NOT_READY_ERROR` 与 `MLX90640_FRAME_INCOMPLETE_ERROR` 的业务层区分：未 ready 或未凑齐双 subpage 不再作为硬错误刷屏，任务层只对真正 I2C/HAL/数据错误输出 capture error。
- 为 `MLX90640_GetFrameData` 增加调试读数接口，记录最后一次状态寄存器、frame/aux 校验失败 index 和失败值，便于后续定位 `0x7FFF` 帧数据问题。

### 调试经验记录
- **不要一开始就把问题归咎于 MLX90640 本体**。本次问题最后分拆为三类：`initTaskBoot` 启动初始化、`MLX90640Task` 运行期采集、`CanBTextTask`/USART3 调试发送等外围路径。只有通过“只保留 defaultTask -> 恢复 initTaskBoot -> 恢复 MLX90640Task -> 恢复 CanBTextTask -> 最后恢复业务发送”的方式，才能确认基础调度器、初始化流程和运行期采集各自是否独立正常。
- **板载 LED 心跳必须肉眼可辨识**。最初 `defaultTask` 使用 `osDelay(10)` 快速翻转，肉眼容易误判为“常亮/卡死”；改为 `osDelay(500)` 后，才能确认 FreeRTOS 基线是否仍在运行。后续调试多任务问题时，务必先准备一个低频心跳基线。
- **`MLX90640_FRAME_DATA_ERROR(-8)` 与 `MLX90640_FRAME_INCOMPLETE_ERROR(-10)` 不是同一种问题**。前者表示 API 校验阶段发现帧/辅助区非法值，后者只是多传感器轮询下单次调用尚未凑齐两个 subpage；如果继续把这两类状态混在一起，会把正常的跨轮询累计误判成硬故障。
- **四路轮询场景下，不应要求一次调用内读齐双子页**。将每个传感器的 subpage 状态独立保存并跨轮询累计后，系统不再因为只读到 `subpage 0` 或 `subpage 1` 就不断报错，也更符合 TCA9548A 多通道复用场景。
- **CAN 发送失败不能再直接进 `Error_Handler()`**。定位过程中发现，开启 `Mlx_SendCanSummary()` 后一旦 `HAL_CAN_AddTxMessage()` 失败（如邮箱满、总线无 ACK、总线状态异常），系统会立即进入全局错误死循环，看起来像“MLX90640 导致卡死”，实际上根因在 CAN1 发送路径。现已改为发送失败直接丢弃当前摘要帧，保住调度器继续运行。
- **USART3 调试通道只适合摘要，不适合长期高密度矩阵输出**。完整 32x24 文本矩阵即使代码逻辑正确，也很容易在小队列 + DMA 发送链路下丢行。当前更适合保留 `Mlx_SendSummary()` 这种单行摘要；矩阵输出只应临时开启，且最好配合更大的调试队列或更低的输出频率。
- **调试发送链路本身也要有失败退出路径**。当前 `App_UART_TransmitDmaBlocking()` 已补充超时和 `HAL_UART_AbortTransmit()` 收尾，避免 USART3 状态卡在 BUSY 时把 `defaultTask` 一并拖死。

### 仍然存在的问题 / 后续待办
- `Mlx_SendCanSummary()` 虽然已不再因 CAN 发送失败导致系统死锁，但 CAN1 发送失败的根本原因（邮箱满、总线接线、ACK 缺失、发送节奏过密或总线状态机问题）仍未完全查清，需要后续结合 `HAL_CAN_GetError()`、邮箱占用状态和实际总线连接继续排查。
- 目前为稳定性考虑，部分任务循环已经临时加入 `osDelay(200)`，这更偏向节流/观察用途，不一定是最终合理的实时参数。后续需根据业务带宽重新收敛任务周期，避免因为人为大延时掩盖潜在竞争条件。
- `defaultTask` 当前更多作为系统基线心跳使用。等 CAN/MLX 运行路径稳定后，应恢复更正式的调试策略，而不是长期依赖 LED 观察系统生存状态。
- `Mlx_SendSummary()` 已恢复，而 `Mlx_SendDebugMatrix()` 仍建议保持注释，仅在需要时临时启用。若未来确实需要长期输出热图原始矩阵，应重新设计 USART3 调试队列深度、发送节奏或改为分包/二进制输出。
- `MLX90640Task` 与其它外围任务（SPI CAN、RS485、publish）目前仍在逐项恢复验证过程中。虽然已确认“默认任务 + 启动任务 + MLX90640 采集 + CanBText 空跑”可以并存，但尚未完成所有任务同时开启下的最终长期稳定性验证。

### 4G / MQTT publish 链路调通
- 完成基于 Y100M 4G 模块标准 AT 固件的 Route-A MQTT 上云路径验证：MCU 端可完成 AT 基础握手、PDP 激活、TCP 连接、MQTT CONNECT、CONNACK 校验，并通过 `AT+CIPSEND` 发布 `fsae/telemetry` 主题下的 TelemetryFrame protobuf 二进制 payload。
- 对齐 PC 验证脚本 `pc_y100m_gps_publish_test.py` 的关键流程：基础 AT 检查后先关闭旧 TCP/PDP 会话，再执行 `CSTT / CIICR / CIFSR` 激活 PDP，之后建立 TCP、发送 MQTT CONNECT、进入 publish；避免此前在拿到 IP 后再 `CIPSHUT` 导致 PDP 被关闭的问题。
- 修复 USART1 AT 响应接收时序：发送 AT 命令或 raw MQTT payload 前先打开 `HAL_UARTEx_ReceiveToIdle_DMA` 接收窗口，并在等待周期内累积多段 idle 响应再匹配 `OK`、`>`、`SEND OK`、`ERROR`，降低 FreeRTOS 下快速 echo/OK 分片导致丢响应的风险。
- 增加 USART3 辅助诊断日志：可输出 AT 响应、TCP 异步状态、MQTT CONNACK 与 SEND OK 的原始 HEX，最终确认 CONNACK 字节为 `20 02 00 00`，publish 阶段 `SEND OK` 正常返回。调试过程中也确认原始日志不能阻塞 USART1 接收关键路径，否则会只收到 echo 的 `AT` 而丢失后续 `OK`。
- 修正 MQTT CONNECT 固定包 Remaining Length：`STM32_DATA_COLLECTION` client id 对应剩余长度为 `0x21`，不再使用错误的 `0x23`，避免 broker 等待不存在的额外字节而不返回 CONNACK。
- Publish4G 任务恢复 MQTT ready 门控：bootstrap 未完成时只输出限频等待日志，不再直接进入 `AT+CIPSEND`；发布失败时加入退避后重新排队，避免失败路径高频重试拖垮串口与模块状态。
- 当前实测成功日志链路：`CONNECT OK` -> `MQTT CONNACK accepted` -> TelemetryFrame protobuf 编码 -> MQTT PUBLISH packet 组包 -> `AT+CIPSEND=30` -> `SEND OK`。后续建议收敛 USART3 高频 HEX 日志，保留少量状态日志后再进行 0.1 秒持续发布稳定性测试。

### 恢复原业务发布路径与 MLX90640 上云验证
- 收敛 4G bring-up 阶段的调试路径：保留短状态日志，关闭原始 HEX 大日志；将 USART3 4G 调试报文、protobuf HEX dump、MQTT publish packet dump 都用编译开关隔离，并在代码注释中说明各调试报文用途和时序影响。
- 恢复 `CanForViehcleTask` 与 `CanForBMSTask` 创建，使 CAN2 整车帧和 CAN1 电池箱帧重新进入各自解码任务，再由解码任务更新共享缓存并向 Publish4G 提交 publish 请求。
- 恢复 `PUBLISH_TOPIC_VEHICLE_STATE` 的正常业务映射：不再使用只带 `motion.gps_speed_kmh` 的 Route-A 最小调试帧，而是回到 `Publish_MapCommonTelemetryFrame()`，从 CANB/BMS/IMU/GPS 等共享缓存编码完整 TelemetryFrame。
- 将 `Publish_MapMotionTelemetry()` 从固定 `gps_speed_kmh = 150` 的调试值恢复为读取 `g_CANB_LoopData.IMU`：根据 `CANB_MOTION_VALID_GPS / ACCEL / GYRO / YAW` 标志填充 GPS 速度、加速度、yaw rate 和 yaw angle。
- 修复 Publish4G 发送任务仍只处理 `PUBLISH_TOPIC_VEHICLE_STATE` 的遗留限制。现在发送任务会对队列中任意合法 topic 调用 `Publish_BuildFrame(item.topic, ...)`，因此 MLX90640 采集任务提交的 `PUBLISH_TOPIC_THERMAL_SUMMARY` 不再被静默丢弃。
- 释放 PD13 作为阶段诊断灯：不再在 CAN2 中断入口翻转，而是在 `0x301` GPS 速度帧真正进入解码分支时翻转；MQTT publish 成功后也翻转一次，用于区分“CAN 中断到了”“解码到了”和“4G 发出去了”。
- 增加轻量 publish 诊断：记录最近一次 payload 长度、发送状态、成功/失败计数，并在失败时输出 `[PUB-DIAG]` 短日志，避免重新开启 HEX 大日志造成时序扰动。
- 通过直接发送 CAN `0x301` 验证 CAN2 -> 解码缓存 -> protobuf -> 4G MQTT publish 整链路可更新云端。此前偶发未更新更可能与调试环境信号、云端刷新或 TCP/MQTT 状态有关，而非主路径逻辑错误。
- 释放 MLX90640 thermal summary 发布后，发现 `payload len=252`、MQTT 总包 `len=271` 超过原 `Y100M_MQTT_TX_BUFFER_SIZE=256`。将 MQTT TX buffer 扩至 `512`，保留超长帧保护，实测 thermal summary 可通过 `AT+CIPSEND` 正常发布。

### MCP2518FD 初始化链路修复
- 修复 MCP2518FD 片选初始电平：`MCP2518_CS_A/B` 低有效，GPIO 初始化阶段应保持空闲高电平，避免上电后芯片被意外选中并误收 SPI 毛刺。
- 修正 MCP2518FD `OSC` 寄存器 ready 位定义：`PLLRDY=bit8`、`OSCRDY=bit10`、`SCLKRDY=bit12`。当前 40MHz 晶振且未启用 PLL 的初始化路径只等待 `OSCRDY`，避免误把正常读值 `OSC=0x00000400` 判定为晶振未就绪。
- 增加 MCP2518FD 初始化分阶段诊断日志：SPI reset、配置模式、OSC、IOCON、ECC、RAM、bit timing、FIFO/filter、normal mode 等步骤失败时输出明确阶段，`SetMode` 超时时打印 `C1CON` 原始读值，OSC 超时时打印 `OSC` 原始读值。
- 排查确认：未给 MCP2518FD 上电时，`C1CON=0xFFFFFFFF` 表示 MISO 悬空/上拉、芯片未响应；芯片上电后可读到 `OSC=0x00000400`，说明 SPI 通讯和晶振起振均正常。
- 为排查 MCP2518FD 启动日志，临时静默 4G publish 等待/失败类调试报文，避免 `[4G-PUB]`、`[PUB-DIAG]` 抢占串口日志视野；保留 MCP 和启动主线日志。
- 当前实测状态：MCP2518FD-A 初始化已成功通过，后续若再出现 `@14 Normal Mode` 类问题，应转向 CAN 收发器使能、总线空闲、终端电阻和波特率匹配排查。

### CANA 硬件滤波实现
- 新增 `MCP2518FD_SetStdFilters()` API，允许每片 MCP2518FD 在初始化前预配置白名单过滤表，过滤配置与 `MCP2518FD_Init()` 解耦。
- `FDCAN_BUS_A` / `SPI1 + CS_A` 配置 20 个 CANA 控制器标准帧 ID 精确过滤，`FDCAN_BUS_B` / `SPI3 + CS_B` 保持全通。
- Filter 寄存器配置策略：`MASK0 = FILTER_MIDE | STD_ID_MASK`（同时精确匹配 IDE 和 11-bit ID），每个 CANA ID 占用一个 `FLTOBJ`，匹配后统一进入 RX FIFO1。
- 增加编译开关 `FDCAN_BUS_A_EXACT_FILTER_ENABLED` 便于测试时一键切换全通/白名单。
- 引入 `APP_SPICANA_READY_CHECK_ENABLED` / `APP_SPICANA_POLL_ENABLED` / `APP_SPICANA_DECODE_ENABLED` / `APP_SPICANA_PUBLISH_ENABLED` 四级逐级隔离开关，用于定位 CANA 运行期故障阶段。

### 业务任务启动门控
- 新增 `App_WaitForBootInit()` 统一门控 helper：除 `defaultTask` 和 `initTaskBoot` 外，所有业务任务（CAN 解码、MLX90640、SPI CAN、Publish4G、VehicleCanTx 等）在进入主循环前等待 `g_bootInitDone == 1U`。
- 此前 `initTaskBoot` 初始化阶段 `osDelay()` 让出 CPU 时，已创建但未门控的业务任务可能提前读写未就绪的外设状态，间接引发 HardFault。加上门控后，初始化阶段的调度延后不再导致任务误运行。

### 任务栈/静态内存优化
- 业务任务静态栈扩容：
  - `MLX90640`：1024 → 1536 uint32_t（4KB → 6KB）
  - `SPICanControl`：512 → 1024 uint32_t（2KB → 4KB）
  - `SPICanCDC`：512 → 1024 uint32_t（2KB → 4KB）
  - `Publish4G`：192 → 1536 uint32_t（768B → 6KB）
  - `initTaskBoot`：1024 → 1536 uint32_t（4KB → 6KB）
- 扩容原因：静态任务栈之间不是共享池而是独立静态数组，扩栈消耗 `.bss` RAM 而非 FreeRTOS heap。`SPICanControl` 运行期涉及 SPI DMA 收发缓冲、MCP FIFO object、CAN 帧副本等，原 512B 栈在 DMA + 解码 + publish 路径下容量不足；`Publish4G` 涉及 protobuf 编码、MQTT 组包和 AT 状态机，原 768B 栈在大包 thermal summary 下可能溢出。

### Publish / 4G 出口路径修复
- `Y100M_MQTT_TX_BUFFER_SIZE` 从 512U 扩至 1024U：`fsae_ThermalSummary_size = 540`，加上 MQTT topic/header 后原 512B 不够覆盖最大 thermal payload 场景。
- `Publish_MapThermalSummary()` 增加固顶数组上限保护：`sensors_count` 和 `chunks_count` clamp 到 ≤4，与 protobuf `sensors[4]` / `chunks[4]` 固定容量对齐，避免后续宏变更导致越界写入。
- 增加 `APP_MLX90640_PUBLISH_ENABLED` 编译开关，便于隔离 thermal summary publish 路径。
- 实测结论：MLX90640 thermal summary publish 触发的 HardFault 并非 CANA 滤波引入，而是 publish 出口资源（Publish4G 栈 + MQTT TX buffer）对大包 payload 余量不足。

### 调试指示灯重新分配
